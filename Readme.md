# Timeshiit

Timeshiit is a .NET command-line utility for preparing Zoho timesheet CSV data using leave/holiday cleanup, Jira-based
enrichment, and validation checks.

## Functionality overview

- Initialize supporting data files (`init`)
    - `init leaves`: pulls leave/holiday data from Zoho CLI and writes `leaves_<year>.json`
    - `init jobs`: command exists, currently no-op in this revision
  - `init dependencies`: downloads Atlassian CLI (`acli.exe`) and Zoho CLI (`zcli.exe`) into the current working
    directory
- Process timesheet CSV files (`timelogs`)
    - `cleanup`: normalizes/removes rows (weekends, leaves/holidays, out-of-range dates)
    - `enrich`: maps Jira data to Zoho client/project/task values
    - `verify`: checks day-level totals and invalid reporting patterns
- Validate fallback rules (`validate rules`)
    - validates external Jira fallback mapping rules JSON

## Requirements

- .NET SDK 10.0.x (project target: `net10.0`)
- `ZohoCLI.exe` available for `init leaves`
- `acli.exe` (Atlassian CLI) available for `timelogs enrich`
- Input files used by commands:
    - `timesheet.csv`
    - `jobs.json`
    - `leaves_<year>.json` (generated by `init leaves`)
    - `jira_fallback_rules.json` (optional for `timelogs enrich`, recommended)

## Build and run

```powershell
dotnet build .\TimeshiitCli\TimeshiitCli.csproj
.\TimeshiitCli\bin\Debug\net10.0\TimeshiitCli.exe -h
```

## Typical usage workflow

1. Initialize leave/holiday data for the year:

```powershell
.\TimeshiitCli\bin\Debug\net10.0\TimeshiitCli.exe init leaves -y 2026 -o .
```

2. Ensure `jobs.json` exists in your working folder. This is currently manual operation to get it from Zoho. API doesn't
   provide all the necessary data.
3. Get the standup or timesheet data from AI agent.
4. Clean up generated timesheet rows:

```powershell
.\TimeshiitCli\bin\Debug\net10.0\TimeshiitCli.exe timelogs cleanup -csv .\timesheet.csv -df dd-MMM-yy
```

5. Verify totals and invalid entries:

```powershell
.\TimeshiitCli\bin\Debug\net10.0\TimeshiitCli.exe timelogs verify -csv .\timesheet.csv -df dd-MMM-yy
```

6. Enrich rows from Jira + fallback rules:

```powershell
.\TimeshiitCli\bin\Debug\net10.0\TimeshiitCli.exe timelogs enrich -csv .\timesheet.csv -j .\jobs.json -r .\jira_fallback_rules.json -df dd-MMM-yy
```

6. When you are not happy with the enriched values, you can tell the agent (or do it manually) to update the fallback
   rules. Then you can verify the rules file syntax by running:

```powershell
.\TimeshiitCli\bin\Debug\net10.0\TimeshiitCli.exe validate rules -r .\jira_fallback_rules.json
```

## External rules engine

Timeshiit supports an external fallback mapping rules engine for Jira-to-Zoho mapping when Jira fields cannot directly
resolve project/task values.

### Related files

- Runtime rules file (default path): `jira_fallback_rules.json`
- Example file: `jira_fallback_rules.example.json`
- JSON schema: `jira_fallback_rules.schema.json`
- Engine implementation: `TimeshiitCli\Commands\Timelogs\JiraFallbackRuleEngine.cs`
- Data model: `TimeshiitCli\Data\JiraFallbackRules.cs`

### When rules are used

In `timelogs enrich`:

1. The tool tries to read Jira fields (`customfield_13650`, `customfield_13651`) and issue type via `acli.exe`.
2. If project/task are resolved from Jira (or Jira parent), Zoho job matching is done via fuzzy matching.
3. If project/task are still unresolved, fallback rules are evaluated against the current row.
4. If no rule matches, row values are set to `ASK BOJAN`.

If the rules file is missing, enrichment continues with an empty rule set (no external fallback mappings).

### Rule structure

Top-level JSON:

- `version` (integer, minimum 1)
- `rules` (ordered array)

Each rule supports:

- `id` (required, unique identifier)
- `description` (optional human-readable text)
- `whenAll` (required AND conditions)
- `whenAny` (optional OR conditions, at least one must match when provided)
- `set` (required; at least one of `clientName`, `projectName`, `taskName`)
- `stopProcessing` (optional, defaults to `true`)

Condition fields:

- `column`: one of `date`, `clientName`, `projectName`, `taskName`, `emailId`, `jiraNo`, `hours`, `comment`, `issueType`
- `operator`: one of `equals`, `contains`, `startsWith`, `endsWith`, `regex`
- `value`: expected value/pattern
- `ignoreCase`: optional boolean (defaults to `true`)

### Evaluation semantics

- Rules are evaluated in order.
- A rule matches only when:
    - all `whenAll` conditions match, and
    - `whenAny` is either absent/empty or at least one `whenAny` condition matches.
- On match, `set` values are applied to the row.
- Processing stops after a match unless `stopProcessing` is explicitly `false`.
- Regex operators use culture-invariant matching with a timeout.

### Rule validation

Use:

```powershell
.\TimeshiitCli\bin\Debug\net10.0\TimeshiitCli.exe validate rules -r .\jira_fallback_rules.json
```

Validation checks include:

- required fields (`id`, `set`, etc.)
- allowed columns/operators
- at least one assignment in `set`
- regex pattern validity
- JSON deserialization/parsing errors

## Notes

- `init jobs` currently completes without generating `jobs.json`; supply this file from your own export/source.
- Default file paths and years shown in help output reflect the current working directory and current year.
